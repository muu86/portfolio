# Express.js 서버를 NextJS로 전환

회사는 아웃소싱 업체가 개발한 **Express.js** 앱을 사용 중이었습니다.
혼자서 모든 파트를 담당하다보니 프로젝트가 커질 수록 코드를 관리하기 버거워졌습니다.
외부 업체와 공동 프로젝트를 진행하는 상황에서 프론트엔드와 백엔드에서 버그가 많이 발생해 개발에 더 체계적이고 구조화된 접근 방식이 필요했습니다.
1개월이라는 짧은 일정 내에 백엔드를 새로 구축해야 하는 부담감이 있었지만, 초기에 체계를 잘 잡아놓으면 향후 안정적인 서비스 제공이 가능하다고 판단하고 **NestJS**로의 전환을 결심했습니다.
팀원들이 전환에 동의한 후에 프로젝트 수행과 NestJS 전환을 병행했습니다.

## 전환 이유

### 모듈화된 아키텍쳐가 필요

Express.js는 단순하고 필요한 기능은 라이브러리를 통해서 혹은, 개발자가 직접 구현해서 확장 가능하므로 장점이 많습니다.
NestJS는 **의존성 주입**, **모듈화된 아키텍처**를 기반으로 기타 서버 개발에 필요한 기능들을 기본적으로 제공하는 것이 장점입니다.

프레임워크 차원에서 아키텍쳐 및 **요청 검증**, **예외 처리**, **테스트** 등 개발에 필요한 기능들을 기본적으로 제공하므로 NestJS로 전환 이후 개발 패턴이 더 단순해졌고, 생산성이 높아졌습니다.

### 타입 안전성

기존 코드의 문제점은 Typescript의 장점을 잘 활용하지 못 한다는 것이었습니다.
요청 파라미터의 타입은 Javascript와 Json 기반의 타입 검증을 Middleware를 통해서 하고 있었습니다.

NestJS 환경에서는 Typescript의 장점을 활용하여 **class transformer**와 **class validator**를 통해 요청 검증을 수행했습니다.

### 보안

디지털 뉴트리션의 사운드필 앱은 무료/유료 회원으로 구분되므로 **역할 기반 인증**을 NestJS의 **Guard** 컴포넌트를 통해 구현했습니다.
