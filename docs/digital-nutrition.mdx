# NextJS 전환

회사는 아웃소싱 업체가 개발한 **Express.js** 애플리케이션을 사용하고 있었습니다. 하지만 혼자서 모든 파트를 관리하다
보니, 프로젝트가 커질수록 코드 관리가 점점 어려워졌습니다.

- **모듈화된 아키텍쳐의 필요성**: NestJS는 **의존성 주입**, **모듈화된 아키텍처**를 기반으로 하므로 모듈 단위로 파일
  구조를 체계적으로 관리할 수 있었고, 개발 패턴을 더 단순화할 수 있었습니다.
- **부가 기능 제공**: 프레임워크 차원에서 **요청 검증**, **예외 처리**, **테스트** 등 개발에 필요한 기능들을 기본적으로
  제공하므로 생산성이 향상되었습니다.
- **타입 안전성**: 기존 소스는 Json 기반으로 요청 타입 검증을 수행하고 Raw Query로 데이터베이스에 접근하므로 코드를
  이해하기 어렵고, Typescript의 장점을 활용하기 힘들었으나, NextJS 도입으로 타입 안전한 코드를 작성할 수 있었습니다.

타 기업과의 협업을 진행하면서, 기존 시스템의 백엔드를 새로 구축해야 하는 큰 부담이 있었습니다. 그럼에도 불구하고,
장기적인 서비스 안정성과 확장성을 고려했을 때, 초기 단계에서 체계를 확립하는 것이 중요하다고 판단했습니다. 전환은
팀원들의 동의를 얻은 후 시작되었으며, 기존 프로젝트와 NestJS 전환 작업을 병행하며 진행했습니다.

개발 기간은 1개월을 목표로 설정했으나, 예상보다 시간이 더 소요되어 **2개월**에 걸쳐 NestJS로의 전환 작업을 성공적으로
완료했고, 그 결과 모듈화된 구조와 테스트 용이성을 갖춘 안정적인 백엔드 시스템을 구축할 수 있었습니다.

## 객체 변환과 검증

- **class transformer** 라이브러리를 사용하여 API 요청 객체를 DTO 객체로 변환하고, DTO 객체를 API 응답 객체로 변환하는
  작업을 단순화했습니다.
- **class validator** 라이브러리를 통해 객체 검증을 수행했습니다.

## 보안

- NestJS **Guard** 컴포넌트를 활용하여 **OAuth**, **JWT** 기반의 역할(Role) 인증 시스템을 구축했습니다. 디지털
  뉴트리션의 사운드필 앱은 무료/유료 회원으로 구분되므로 권한 부여를 동적으로 처리했습니다.

## ORM 도입

- Raw Query를 사용하는 기존 소스의 타입 안정성을 강화하기 위해 ORM을 도입했습니다.
- **drizzle**로 개발을 시작했으나, 단순 쿼리 빌더의 역할만 하는 것에 한계를 느꼈습니다.
- **micro-orm**: Java JPA와 유사한 **변경 추적**, **Cache** 등의 기능을 제공하는 micro-orm을 최종적으로 선택했습니다.
- **데이터 마이그레이션**: 기존 데이터 스키마를 많이 변경했기 때문에 운영 중인 데이터를 마이그레이션 하는 것이 큰
  과제였습니다. Flyway, Liquibase 등의 마이그레이션 툴을 조사했으나, 데이터베이스 버전 관리가 주 용도이고, 데이터를
  이전하는 용도로 사용하기는 어렵다고 판단했습니다. 마이그레이션 용으로 별도의 Node.js 프로젝트를 개발하여
  마이그레이션을 수행했습니다. 기존 데이터를 변경된 스키마대로 변경하여 insert하는 반수동 작업이었습니다.

## 구독/결제

- 결제 이후 스토어에서 구독이 변경/취소/갱신 되었을 때 처리하는 로직이 없었기 때문에, 스토어로부터 **실시간 알림**을
  받아 처리하는 모듈을 새로 개발했습니다.
- **ios**: **app-store-server-library-node**을 사용했습니다.
- **android**: **Google Cloud Pub/Sub**을 사용했습니다.
- **web**: **Port One** 으로부터 **billing key**를 발급 받아 정기 결제를 처리했습니다.
- 스토어 및 포트원에 **AWS API Gateway** 도메인을 공개하고 **AWS Lambda**로 실시간 알림을 백엔드 서버로 전송했습니다.

## 테스트 자동화

작은 조직일수록 자동화가 더욱 핵심적입니다. 그 중에서도 테스트 자동화는 코드 품질을 높이고 유지보수를 용이하게 만드는
핵심 요소입니다. 그러나 시간에 쫓겨, 모든 기능에 대한 테스트 코드를 작성하기는 어려운 상황이었습니다.

그럼에도 불구하고, **구독/결제 파트**만큼은 테스트 코드가 반드시 필요하다고 생각해서 **Jest**, **Test Container** 로
단위 테스트, 통합 테스트를 자동화했습니다.

## Terraform 도입

- 클라우드 인프라를 쉽게 관리하기 위해서 Terraform을 도입했습니다.

- **명확한 인프라 정의**: 인프라 구성을 코드로 명시하여, 팀원 누구나 전체 아키텍처를 한눈에 파악할 수 있었습니다. 버전
  관리 시스템과의 통합을 통해 변경 내역을 추적하고, 롤백도 손쉽게 수행할 수 있었습니다.

- **팀원 간 협업 강화**: IaC(Infrastructure as Code) 도입으로 인해 모든 인프라 설정이 코드에 기록되어 있으므로,
  개발자들이 인프라 구조를 쉽게 이해하고 공유할 수 있었습니다.

- **모듈화된 재사용성**: Terraform의 Module 기능을 적극 활용하여, 여러 환경(개발, 스테이징, 프로덕션 등)에 동일한 AWS
  리소스를 일관되게 프로비저닝했습니다.

## AWS EKS

### 오토스케일링 문제

기존 백엔드는 Docker Swarm 환경에서 운영되고 있었습니다. 그러나 Docker Swarm은 기본적으로 오토스케일링 기능을 제공하지
않아, 특히 우리 서비스보다 훨씬 높은 트래픽을 처리하는 협력 기업들이 있는 상황에서 **확장성**에 대한 우려가
제기되었습니다. 이에 따라, Docker 컨테이너 환경에서 오토스케일링을 구현하기 위해 ECS Fargate와 EKS를 검토했습니다.

- **ECS Fargate**는 관리형 서버리스 컨테이너 서비스로, 운영이 간편하여 스타트업 환경에 적합해 보였습니다. 그러나 EC2
  기반 인프라에 비해 ECS의 요금이 다소 높게 책정되는 점이 비용 측면에서 부담으로 작용할 수 있었습니다.

- 반면, **EKS**는 클러스터 관리에 필요한 **고정 비용**만으로 운영이 가능해, 장기적으로 볼 때 비용 효율성을 기대할 수
  있어서 선택했습니다.

### Karpenter로 오토스케일링 해결

- Karpenter는 AWS에서 제공하는 클러스터 오토스케일링 솔루션이므로 AWS EKS에서 사용하기 적합하다고 판단했습니다.
- Kubernetes 클러스터 내에서 실행되는 Karpenter Controller Pod를 통해 스케쥴링되지 않은 Pod를 감시하고 필요한
  Node(EC2)를 자동으로 프로비저닝합니다.
- 테스트 결과 2분 ~ 3분 내외로 Node를 프로비저닝하고 5분 안에 Pod를 구동할 수 있었습니다.
- AWS Autoscailing Group은 직접 사용해보진 않았으나, CloudWatch 기반으로 동작하므로 클러스터 내에서 동작하는 Karpenter에
  비해 노드 프로비저닝 속도가 느리다는 걸 확인하여 Karpenter로 결정했습니다.

### 하나의 Kubernetes 클러스터 안에서 여러 환경 배포

- 여러 기업과의 협업 과정에서, 운영 환경에 배포하기 전 각 개발 중인 기능(feature)에 대해 **별도의 API 엔드포인트를
  제공**할 필요가 있었습니다.
- Kubernetes와 Terraform의 **모듈** 기능을 활용하여 동일한 AWS 리소스를 여러 환경에 배포하고, 서로 격리된 환경에서
  테스트할 수 있도록 설정하였습니다. 이를 통해 **비용을 최적화**하면서 **각 환경의 독립성**을 확보했습니다.

### Kubernetes 도입 시 겪은 도전 과제

- **학습 비용**: 팀 내 동료 개발자들이 합류하면서 학습 부담이 발생했습니다.

- **기술 부채**: 퇴사 전, 팀 내에서 EKS 환경을 유지할지 아니면 EC2와 EC2 Auto Scaling 환경으로 전환할지에 대해 논의가
  있었습니다. 저는 EC2 Auto Scaling으로 다시 전환할 것을 제안했으나, 팀원들의 의견을 존중하여 EKS 환경 유지를
  선택했습니다. 상세한 인수인계 문서를 작성하여 이후 팀이 클러스터를 효율적으로 관리할 수 있도록 했습니다. 문서에는
  업데이트 과정과 버전 관리 방법 및 제가 겪었던 오류들에 대한 상세한 설명을 포함시켜, 후속 작업이 원활하게 이루어지도록
  했습니다.

- **버전 관리**: EKS는 구버전 Kubernetes를 사용할 경우 추가 요금이 발생할 수 있어 버전 관리가 중요했습니다. Karpenter
  1.0.0 버전이 정식 릴리즈된 시점에 맞춰, Kubernetes 클러스터와 Karpenter의 버전 업데이트를 성공적으로 진행했습니다.
  버전 업그레이드를 통해 시스템의 성능 안정성과 비용 최적화가 개선되었고, 향후 기술 부채를 줄일 수 있는 기반을
  마련했습니다.

# CI/CD 자동화
