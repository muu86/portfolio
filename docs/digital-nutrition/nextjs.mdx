# Express.js 서버를 NextJS로 전환

회사는 아웃소싱 업체가 개발한 **Express.js** 애플리케이션을 사용하고 있었습니다. 하지만 혼자서 모든 파트를 관리하다
보니, 프로젝트가 커질수록 코드 관리가 점점 어려워졌습니다. 특히 외부 업체와 협업하는 상황에서 프론트엔드와 백엔드에서
발생하는 버그가 많아졌고, 이에 따라 개발 과정에 더 체계적이고 구조화된 접근 방식이 필요함을 절감했습니다. 1개월이라는
짧은 기간 내에 백엔드를 새로 구축해야 하는 부담이 있었지만, 장기적으로 안정적인 서비스를 제공하려면 초기 단계에서 체계를
확립하는 것이 중요하다고 판단했습니다. 이에 따라 NestJS로의 전환을 결심하고, 팀원들의 동의를 얻은 후, 기존 프로젝트와
NestJS 전환 작업을 동시에 진행했습니다.

## 전환 이유

### 모듈화된 아키텍쳐가 필요

Express.js는 단순하고 필요한 기능은 라이브러리를 통해서 혹은, 개발자가 직접 구현해서 확장 가능하므로 장점이 많습니다.
NestJS는 **의존성 주입**, **모듈화된 아키텍처**를 기반으로 기타 서버 개발에 필요한 기능들을 기본적으로 제공하는 것이
장점입니다.

프레임워크 차원에서 아키텍쳐 및 **요청 검증**, **예외 처리**, **테스트** 등 개발에 필요한 기능들을 기본적으로 제공하므로
NestJS로 전환 이후 개발 패턴이 더 단순해졌고, 생산성이 높아졌습니다.

### 타입 안전성

기존 코드의 문제점은 Typescript의 장점을 잘 활용하지 못 한다는 것이었습니다. 요청 파라미터의 타입은 Javascript와 Json
기반의 타입 검증을 Middleware를 통해서 하고 있었습니다.

NestJS 환경에서는 Typescript의 장점을 활용하여 **class transformer**와 **class validator**를 통해 요청 검증을
수행했습니다.

### 보안

디지털 뉴트리션의 사운드필 앱은 무료/유료 회원으로 구분되므로 **역할 기반 인증**을 NestJS의 **Guard** 컴포넌트를 통해
구현했습니다.

# AWS 인프라 구축

## Terraform 도입

- 클라우드 인프라를 쉽게 관리하기 위해서 Terraform을 도입했습니다.

- **명확한 인프라 정의**: 인프라 구성을 코드로 명시하여, 팀원 누구나 전체 아키텍처를 한눈에 파악할 수 있었습니다. 버전
  관리 시스템과의 통합을 통해 변경 내역을 추적하고, 롤백도 손쉽게 수행할 수 있었습니다.

- **팀원 간 협업 강화**: IaC(Infrastructure as Code) 도입으로 인해 모든 인프라 설정이 코드에 기록되어 있으므로,
  개발자들이 인프라 구조를 쉽게 이해하고 공유할 수 있었습니다.

- **모듈화된 재사용성**: Terraform의 Module 기능을 적극 활용하여, 여러 환경(개발, 스테이징, 프로덕션 등)에 동일한 AWS
  리소스를 일관되게 프로비저닝했습니다.

## AWS EKS

### 오토스케일링 문제

기존 백엔드는 Docker Swarm 환경에서 운영되고 있었습니다. 그러나 Docker Swarm은 기본적으로 오토스케일링 기능을 제공하지
않아, 특히 우리 서비스보다 훨씬 높은 트래픽을 처리하는 협력 기업들이 있는 상황에서 확장성에 대한 우려가 제기되었습니다.

이에 따라, Docker 컨테이너 환경에서 오토스케일링을 구현하기 위해 AWS ECS Fargate와 AWS EKS를 면밀히 검토했습니다. ECS
Fargate는 관리형 서버리스 컨테이너 서비스로, 운영이 간편하여 스타트업 환경에 적합해 보였습니다. 그러나 EC2 기반 인프라에
비해 ECS의 요금이 다소 높게 책정되는 점이 비용 측면에서 부담으로 작용할 수 있었습니다. 반면, EKS는 클러스터 관리에
필요한 고정 비용만으로 운영이 가능해, 장기적으로 볼 때 비용 효율성을 기대할 수 있었습니다.

### Karpenter로 오토스케일링 해결

- Karpenter는 Kubernetes 클러스터 내에 Karpenter Controller Pod를 실행하고 스케쥴링되지 않은 Pod를 감시하고 필요한
  Node(EC2)를 자동으로 프로비저닝합니다.
- Karpenter의 장점은 오토스케일링 시간이 **2분 ~ 5분** 내외로 빠르다는 것입니다.
- AWS Autoscailing Group은 직접 사용해보진 않았으나, 조사 결과 Autoscailing Group은 CloudWatch 기반으로 동작하므로
  클러스터 내에서 동작하는 Karpenter에 비해 느리다는 걸 확인하여 Karpenter로 결정했습니다.

### 하나의 Kubernetes 클러스터 안에서 여러 환경 배포

- 여러 기업과의 협업 과정에서, 운영 환경에 배포하기 전 각 개발 중인 기능(feature)에 대해 별도의 API 엔드포인트를 제공할
  필요가 있었습니다.
- Kubernetes와 Terraform의 **모듈** 기능을 활용하여 동일한 AWS 리소스를 여러 환경에 배포하고, 서로 격리된 환경에서
  테스트할 수 있도록 설정하였습니다. 이를 통해 **비용을 최적화**하면서 **각 환경의 독립성**을 확보했습니다.

### Kubernetes 도입 시 겪은 도전 과제

- **학습 비용**: 초기에 백엔드 개발자로 혼자 작업할 때는 문제가 없었으나, 팀 내 동료 개발자들이 합류하면서 Kubernetes에
  대한 학습 부담이 발생했습니다.

- **기술 선택의 갈등**: 저는 EKS 대신 EC2와 EC2 Auto Scaling 환경으로 전환하는 방안을 제안하기도 했으나, 팀원들은 EKS
  환경을 유지하길 원했습니다.

- 퇴사 전, Karpenter가 정식 1.0.0 버전으로 릴리즈된 시점을 맞추어 Kubernetes 클러스터와 Karpenter 업데이트를
  완료하였으며, 이후 상세한 인수인계 문서를 작성하여 팀 내 지식 이전을 원활하게 진행하였습니다.
